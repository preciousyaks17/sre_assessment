# instrumentation/1.3-trace-validation-checklist.md

These steps require a live cluster + Kibana access and must be done by hand
once `generate-checkout-traffic.sh` has been run against the real
environment. Screenshots go in `docs/screenshots/` and get referenced from
the main README.

- [ ] Run `./generate-checkout-traffic.sh` with `FRONTEND_URL` pointed at
      the assessment cluster's frontend service/ingress.
- [ ] Kibana -> Observability -> APM -> Services: confirm `frontend`,
      `recommendationservice`, `paymentservice` (and any auto-discovered
      downstream services they call) all show up with recent transactions.
      -> screenshot: `apm-services-list.png`
- [ ] Kibana -> Observability -> APM -> Traces: open the most recent trace
      rooted at `frontend`. Confirm the waterfall shows spans from all
      hops the checkout flow touches.
      -> screenshot: `trace-waterfall-full-checkout.png`
- [ ] Kibana -> Observability -> APM -> Service Map: confirm the call
      graph matches the expected architecture (frontend -> cart/product/
      checkout -> payment/shipping/currency/email).
      -> screenshot: `service-map.png`
- [ ] Verify W3C `traceparent` propagation held across at least 3 hops —
      i.e. all spans in the waterfall share the same trace ID. If any
      service's spans appear as a *separate* trace, that's a broken
      propagation link — note which hop and why (see "known gaps" below).
- [ ] Open the failed checkout transaction (from the bad-card request in
      the script) in APM -> Traces, click into the Error tab, confirm
      exception type/message/stack trace are visible.
      -> screenshot: `error-trace-detail.png`

## Known / expected gaps (requirement 1.3.4: identify + propose fixes)

Document actual findings here once run against the live cluster. Likely
candidates based on the Online Boutique architecture, to check for:

- **shippingservice / currencyservice / emailservice are not in the
  3-service instrumentation set** (only frontend, recommendationservice,
  paymentservice were instrumented per the Section 1.2 scope decision).
  If checkout calls these, they will NOT appear as spans — the trace will
  show a gap between checkoutservice and where the trace context is
  actually captured. **Fix:** these services would need the same
  auto-instrumentation pattern applied (Go/C++/Ruby depending on
  language) to close the gap — noted as a follow-up given time
  constraints, not a bug in what was built.
- **checkoutservice itself** is the orchestrator for the whole flow but
  was not explicitly in the chosen 3 — if it's uninstrumented, the trace
  may show frontend calling payment directly with a context gap where
  checkoutservice's own processing time isn't visible. Same fix as above.
